# 模拟面试战报首题复盘归档包设计

## 背景

首题复盘验收卡已经能判断复盘记录是否可用，但验收结果还没有被整理成下一轮训练可以直接携带的输入。用户完成首题复盘后，如果只看到下一轮训练建议，仍然需要自己从评分、证据、阻断和下一题动作里重新摘重点，容易丢失刚刚形成的复盘价值。

## 目标

在“首题复盘验收卡”和“下一轮训练建议”之间增加“首题复盘归档包”。归档包把首题复盘验收结果整理成 4 个可复用片段：分数快照、证据归档、阻断结论、下一题种子。所有内容继续由本地规则派生，不依赖付费接口、后端状态或第三方服务。

## 方案

首题复盘归档包由 `buildPracticeSessionFirstQuestionReviewAcceptance` 和下一轮题组共同派生：

- 复盘验收为空时，显示等待归档状态。
- 复盘验收处于回修状态时，显示“回修复盘归档包”，强调先保存阻断修复证据。
- 复盘验收处于 ready 状态时，显示“首题复盘归档包”，强调把首题样本作为下一轮训练输入。
- 每个归档项包含来源、归档内容、下一轮用途和丢失风险。
- 主行动复用复盘验收入口，避免引入新的路由和持久化状态。

## 用户价值

归档包把一次复盘从“看完就结束”变成“下一轮训练的种子”。它让用户清楚知道哪些证据已经进入下一轮，哪些内容一旦遗漏会导致训练失焦，从而提升免费本地训练闭环的可持续性。

## 验收

- Markdown 导出包含“首题复盘归档包”，且位于“首题复盘验收卡”之后、“下一轮训练建议”之前。
- 前端战报展示 4 个归档项：分数快照、证据归档、阻断结论、下一题种子。
- 空会话显示等待状态，不出现 `undefined`。
- 聚焦测试、全量测试、构建和 `git diff --check` 通过。
