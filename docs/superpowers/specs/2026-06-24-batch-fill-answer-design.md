# 后台批量补答案设计

## 背景

管理后台现有“流式补答案”通过 SSE 逐题运行，分类留空时会从所有分类中取候选题，但仍受 `count` 限制，最多一次处理 20 道。它适合观察单题生成过程，不适合把几千道待补草稿放到后台持续运行。

## 目标

新增后台批量补答案能力：管理员可启动一个异步任务，默认补齐所有分类下 `DRAFT` 且答案为空的题目，也可按分类限制范围；任务不依赖页面长连接，前端通过轮询查看进度。

## 范围

- 支持启动批量补答案任务。
- 支持查询批量补答案进度。
- 支持分类筛选、API 调用间隔、本次最大处理数量。
- 默认最大处理数量为空，表示跑到当前待补草稿清零。
- 复用现有答案质量门禁、三次重试、结构化字段写入逻辑。
- 保留现有 SSE 流式补答案，用于小批量观察。

## 非目标

- 不新增持久化任务表，服务重启后任务状态不恢复。
- 不实现暂停、继续、取消。
- 不并发调用 AI，避免限流和质量问题。
- 不自动发布草稿，仍由现有草稿审核流程控制发布。

## 后端设计

新增 `BatchFillAnswerRunner`，形态参考 `BatchGenerationRunner`：

- 单线程后台执行。
- 使用 `AtomicBoolean` 防止重复启动。
- 使用内存 `BatchProgressVO` 记录状态。
- 启动时统计待补总量。
- 每次按 ID 升序分页取一批待补草稿，逐题补答案。
- 每题之间按 `delaySeconds` 休眠。
- 成功更新题目内容字段，失败记录错误并继续下一题。

`AiQuestionService` 抽出可同步复用的补单题方法：

- 输入 `Question`。
- 加载分类和标签上下文。
- 构建补答案 prompt。
- 调用现有流式 AI 客户端的静默心跳模式。
- 解析结构化答案。
- 使用 `AiAnswerQualityPolicy` 做质量校验。
- 通过 `questionMapper.updateById` 写回答案字段。
- 返回成功、质量分、错误信息。

现有 `streamFillAnswer` 改为复用该补单题方法，只保留 SSE 进度推送和结果汇总，避免批量任务复制质量门禁逻辑。

## API 设计

新增接口：

- `POST /api/admin/ai/fill-answer-batch`
- `GET /api/admin/ai/fill-answer-batch/status`

启动请求字段：

- `categoryId?: Long`，为空表示所有分类。
- `maxQuestions?: Integer`，为空表示处理全部待补草稿。
- `delaySeconds?: Integer`，默认 3 秒。

响应沿用现有 `ApiResponse` 格式。

## 前端设计

在后台 AI 页面“流式补答案”区域增加后台批量补答案表单：

- 分类选择，留空表示所有分类。
- 本次最大处理数量，留空表示补到清零。
- API 调用间隔秒数。
- 启动按钮。
- 进度卡片显示运行状态、完成题数、总题数、失败数、当前分类/题目和错误列表。

保留现有逐题流式补答案表单，并调整文案，说明它是“小批量观察模式”。

## 测试策略

- Controller 测试：AI 未配置拒绝启动、重复任务返回 409、启动参数正确传递。
- Runner 测试：统计全量待补题、按分类筛选、最大处理数量生效、成功失败计数正确。
- 前端构建：运行 TypeScript 检查，确保新增 API 和类型一致。
