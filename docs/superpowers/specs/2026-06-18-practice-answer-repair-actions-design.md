# 模拟面试回答结构缺口一键修复设计

## 背景

`/practice` 已经具备答题脚手架和实时结构检查。用户现在能知道回答是否缺少结论、机制、场景或边界，但下一步仍要手动回到输入框补内容。真实训练里，反馈必须尽量贴近动作：发现缺口后，应立即给出“按这个结构重答”的可编辑草稿，让用户用最少摩擦修正表达。

本阶段增加“结构缺口一键修复”。它读取实时结构检查结果，为最优先缺口生成重答动作，并支持一键带入回答框。这样 `/practice` 形成提交前闭环：脚手架起稿 -> 实时检查 -> 一键修复 -> 提交评分。

## 目标

- 根据当前回答和实时结构检查结果生成一个主修复动作。
- 修复动作覆盖四类缺口：补结论、补机制、补场景、补边界。
- 如果回答已完整，动作变为“压缩成 60 秒版本”，继续提升口述质量。
- 点击修复动作后，把保留原回答的重答模板写回回答框。
- 面板保留提交自由，不强制用户必须点击修复动作。
- 纯函数输出不得出现 `undefined`。

## 非目标

- 不改正式评分算法。
- 不新增后端接口，不调用外部 AI。
- 不保存修复动作历史。
- 不自动覆盖用户输入；必须由用户点击后才带入回答框。

## 修复动作规则

优先级按实时结构检查的缺口顺序：

1. 缺结论：生成“补第一句结论”动作。
2. 缺机制：生成“补核心机制”动作。
3. 缺场景：生成“补项目场景”动作。
4. 缺边界：生成“补边界风险”动作。
5. 四段完整：生成“压缩 60 秒版”动作。

动作模板示例：

```markdown
原回答保留：
HashMap 多线程不安全。机制：扩容和 put 时可能发生覆盖写。

请按下面结构重答：
结论：
我会先明确回答「HashMap 为什么线程不安全？扩容时会发生什么？」：...

机制：
...

场景：
请补一个线上/项目场景，说明触发条件、处理动作和验证指标。

边界：
...
```

## UI 设计

- 在“回答结构实时检查”面板中增加主按钮。
- 按钮文案随动作变化，例如“补项目场景”“补边界风险”“压缩 60 秒版”。
- 点击后由父级 `Practice` 写入回答框，并清空旧评分。
- 面板标题、分数和四段状态保持不变。
- 移动端按钮占满可用宽度，避免挤压维度条目。

## 测试策略

- 新增 `practiceAnswerRepairAction.test.ts`：
  - 部分回答缺场景时生成 `scenario` 动作，模板保留原回答并提示补项目场景。
  - 空回答生成 `conclusion` 动作，模板包含题目标题。
  - 完整回答生成 `compress` 动作，模板提示压缩成 60 秒版本。
  - 所有输出不包含 `undefined`。
- 扩展 `PracticeAnswerReadinessPanel.test.tsx`：
  - 缺场景时渲染“补项目场景”按钮。
  - 点击按钮后回调收到修复模板。
