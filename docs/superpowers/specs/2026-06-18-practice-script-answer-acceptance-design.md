# 模拟面试脚本追问回答验收设计

## 背景

`/practice` 已经具备本题面试官脚本：系统能根据当前题历史评分生成三步追问，并把单个追问带入回答框。当前缺口是：用户开始回答追问后，只能看到通用“回答结构实时检查”，无法判断这段回答是否真正回应了当前脚本追问。

真实面试追问通常不是重新背一遍主答案，而是围绕面试官给出的压力点补证据、补边界、补取舍。本阶段新增“追问回答验收”，在回答框下方实时识别当前追问草稿，并给出是否可提交的验收结论，让用户免费获得更接近真人面试官的二次检查。

## 目标

- 在 `/practice` 回答框下方新增“追问回答验收”面板。
- 仅当回答草稿包含 `追问：...` 时展示有效验收，否则给出待追问状态。
- 根据当前题的本题面试官脚本匹配草稿中的追问 prompt。
- 输出追问回答的：
  - 分数
  - 状态
  - 总结
  - 下一步动作
  - 验收项列表
- 验收项覆盖：
  - 是否直接回应追问
  - 是否补充可追问证据
  - 是否覆盖该脚本步骤的压力点
  - 是否命中脚本维度要求
- 所有逻辑前端本地完成，完全免费，不调用后端或外部 AI。

## 非目标

- 不替换现有“回答结构实时检查”。通用主答案检查仍然保留。
- 不修改评分提交接口，也不改变 `InterviewAttempt` 存储结构。
- 不新增计时器、语音、录音或多轮聊天状态。
- 不要求用户显式选择脚本步骤；系统从草稿追问文本自动匹配。

## 规则

### 草稿解析

- 支持格式：`追问：<prompt>\n\n我的回答：<answer>`。
- 未出现 `追问：` 时，面板状态为 `idle`。
- 出现追问但没有回答正文时，状态为 `empty`。
- 追问文本会与当前脚本步骤 prompt 做归一化包含匹配。
- 无法匹配到脚本步骤时，仍按通用追问验收，提示“先从本题面试官脚本带入追问更精准”。

### 验收项

- `direct`：回答正文需要直接回应追问主题。命中题目关键词、脚本 prompt 关键词或出现结论类表达即通过。
- `evidence`：需要提供可被继续追问的证据。项目、线上、指标、数据、复盘、验证等信号命中即通过。
- `pressure`：需要回应脚本压力点。压力点关键词、失败/边界/反问/权衡等信号命中即通过。
- `criterion`：需要满足脚本维度要求：
  - `coverage`：定义、机制、原理、替代方案。
  - `structure`：结论、原因、场景、边界、顺序。
  - `specificity`：项目、线上、指标、触发条件、验证。
  - `risk`：风险、边界、失败、兜底、降级。
  - `advanced`：方案、对比、取舍、选择、优缺点。

### 分数和状态

- 每个验收项 25 分。
- `idle`：非追问草稿，0 分。
- `empty`：已识别追问但没有回答正文，0 分。
- `draft`：25 到 49 分。
- `ready`：50 到 74 分。
- `passed`：75 分及以上。

### 下一步动作

- 非追问草稿：提示先从脚本选择追问。
- 空回答：提示先写出追问回答。
- 缺直接回应：提示先正面回答面试官问题。
- 缺证据：提示补项目动作、指标或验证方式。
- 缺压力点：提示回应脚本压力点。
- 缺维度：提示补齐当前维度要求。
- 全部通过：提示可提交评分。

## UI 设计

- 面板放在 `PracticeAnswerReadinessPanel` 下方，提交按钮上方。
- 使用紧凑横向布局：左侧分数和状态，右侧总结、下一步动作和四个验收项。
- `idle` 和 `empty` 使用浅黄提示；`ready` 使用浅蓝；`passed` 使用浅绿。
- 文案强调“追问验收”，避免与主答案结构检查混淆。
- 移动端复用现有回答检查面板的单列布局。

## 测试策略

- 新增 `practiceScriptAnswerAcceptance.test.ts`：
  - 非追问草稿返回 `idle`。
  - 从脚本 prompt 带入但未回答返回 `empty`。
  - 回答包含项目指标、压力点和维度特征时返回 `passed`。
  - 只正面回答但缺证据时优先提示补证据。
  - 无法匹配脚本步骤时仍返回通用追问验收，不出现 `undefined`。
- 新增 `PracticeScriptAnswerAcceptancePanel.test.tsx`：
  - 渲染非追问提示。
  - 渲染通过状态和 100 分。
  - 对部分回答展示下一步动作。
