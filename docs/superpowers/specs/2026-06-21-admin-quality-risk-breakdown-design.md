# 管理后台分类风险拆解设计

## 背景

下一阶段重点是 P2“内容质量超过题量焦虑”。当前管理后台已经有质量总览、分类风险排行、草稿筛选和发布门禁，但分类排行只展示部分缺口，运营人员仍需要猜测“这个分类为什么风险高、应该先修哪个模块”。这会拖慢从 DRAFT 到 PUBLISHED 的治理闭环。

## 目标

- 在分类风险排行中暴露完整结构缺口：摘要、原理、对比、场景、风险、项目、结构段、代码示例、难度异常。
- 为每个分类计算最优先处理的风险项，前端展示“首要缺口”和可直达草稿筛选的入口。
- 继续复用现有 `/admin/draft-review` 风险筛选，不引入新表、新后端路由、账号系统或第三方服务。

## 非目标

- 不改 AI 供应商、模型、密钥或生成成本策略。
- 不扩大题库数量，不重写初始化 SQL。
- 不改变发布门禁阈值，只提高后台看板的解释力和行动入口。

## 设计

后端扩展 `AdminCategoryQualityVO`，把 `AdminQualityService.CategoryStats` 已经计算但未返回的风险计数补全到 API 响应中。这样前端和运营后台可以使用同一套统计口径，不复制后端质量判断。

前端新增一个小的 `qualityRisk` 工具，把分类质量行转成有序风险项。排序优先级先按数量降序，再按发布阻断强度排序：空答案、短答案、缺结构段、难度异常优先于普通模块缺口。`Dashboard` 使用该工具展示完整风险标签，并给首要缺口提供“处理”按钮，跳到 `/admin/draft-review?categoryId=...&risk=...`。

## 验收

- 后端测试证明完整风险计数会出现在 `AdminCategoryQualityVO`。
- 前端工具测试证明首要缺口和草稿筛选 URL 可预测。
- 后台总览测试证明分类风险排行能展示新增缺口和首要处理入口。
- 运行聚焦前后端测试，至少覆盖 `AdminQualityServiceTest`、`Dashboard.test.tsx` 和新增前端工具测试。
