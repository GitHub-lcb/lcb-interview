# 本轮证据缺口设计

## 背景

战报已经能给出面试官决策、行动优先级和下一轮训练，但用户仍然需要自己判断“这题为什么不像真实项目经验”。面试中真正卡人的不是分数本身，而是证据缺口：缺少场景、指标、边界、取舍或风险预案。战报应该把低分维度翻译成可被追问的证据缺口，让用户知道下一句要补什么。

## 目标

- 在本轮战报新增“本轮证据缺口”。
- 从当前队列每题最新模拟反馈中提取低分维度。
- 每个缺口包含题目、低分维度、面试官追问、修复提示和入口。
- 优先展示最危险的 3 个缺口，避免信息过载。
- Markdown 导出和面板展示保持一致。
- 不新增后端接口，不依赖外部服务，继续保持完全免费。

## 非目标

- 不替代薄弱能力雷达；雷达看维度均值，证据缺口看具体题目上的可追问漏洞。
- 不做自然语言大模型生成，先用本地规则保证离线可用。
- 不记录用户是否已经修复缺口，后续可由复测结果自然更新。

## 方案

新增 `PracticeSessionEvidenceGaps` 类型和 `buildPracticeSessionEvidenceGaps(queue, progress)` 构建器。

识别规则：

1. 只读取当前队列内每题最新一次模拟反馈。
2. 低于 75 分的评分维度进入候选缺口。
3. 分数越低、题目总分越低、维度越偏“场景细节/风险边界”，排序越靠前。
4. 每题最多保留最危险的两个维度，整体最多展示三个缺口。
5. 没有低分维度时展示“证据链暂稳”，提示用户继续复测。

维度映射：

- 知识覆盖：追问遗漏链路、关键条件和必要步骤。
- 表达结构：追问是否能按背景、方案、结果重讲。
- 场景细节：追问项目规模、指标、数据和个人职责。
- 风险边界：追问失败边界、降级策略和取舍依据。

面板位置放在“本轮行动优先级”之后、“下一轮训练”之前：先告诉用户执行顺序，再说明最该修补的证据。

## 测试策略

- 工具测试覆盖低分场景导出“本轮证据缺口”、追问和修复提示。
- 空队列/未答题场景导出等待提示。
- 面板测试覆盖缺口列表和主行动导航。
- 提交前运行 focused tests、完整前端测试、生产构建和 `git diff --check`。
