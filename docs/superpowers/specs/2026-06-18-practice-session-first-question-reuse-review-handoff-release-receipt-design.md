# 模拟面试战报首题复用复盘回流放行回执设计

## 背景

首题复用复盘回流放行门禁已经能判断回流动作是否可以进入下一轮训练，但门禁结果仍停留在页面提示层面。用户如果要复盘当天训练、交接给下一轮首题，仍需要自己把放行状态、阻断项、证据和下一步入口整理成一份可复制的记录。缺少放行回执会让闭环在“能不能走”之后缺少“带着什么走”的交接凭据。

## 目标

在“首题复用复盘回流放行门禁”和“下一轮训练建议”之间新增“首题复用复盘回流放行回执”。回执把门禁裁决压缩成 4 个可执行交接项：放行结论、随身证据、阻断处理和下一轮入口。所有内容从当前战报上下文派生，不依赖后端状态、账号系统、付费功能或第三方服务。

## 方案

放行回执由 `buildPracticeSessionFirstQuestionReuseReviewHandoffReleaseGate` 派生：

- 门禁为空时，显示“等待首题复用复盘回流放行回执”，提示先完成回流放行门禁。
- 门禁阻断时，显示“回流回执待修复”，主行动回到回流修复入口。
- 门禁通过时，显示“回流回执已生成”，主行动进入下一轮首题。
- 每个回执项包含结论、交接证据、下一步动作和状态，保证用户能复制到笔记或下一轮训练说明。
- 回执只复用门禁入口，不新增路由、存储或服务端状态。

## 用户价值

放行回执把“系统判断可以/不可以进入下一轮”转成“下一轮应该携带哪些证据、先处理哪个阻断、点哪里继续”的轻量交接单。用户不需要在长战报里重复翻找门禁证据，也能把当前训练闭环自然带到下一轮首题。

## 验收

- Markdown 导出包含“首题复用复盘回流放行回执”，且位于“首题复用复盘回流放行门禁”之后、“下一轮训练建议”之前。
- 前端战报展示 4 个回执项：放行结论、随身证据、阻断处理、下一轮入口。
- 每个回执项展示“交接证据”“下一步动作”“状态”。
- 空会话显示等待状态，不出现 `undefined`。
- 聚焦测试、全量测试、构建和 `git diff --check` 通过。
