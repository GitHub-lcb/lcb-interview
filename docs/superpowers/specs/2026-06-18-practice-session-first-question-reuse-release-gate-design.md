# 模拟面试战报首题复用放行门禁设计

## 背景

首题复用回执验收卡已经能检查分数、证据、阻断和下一题接续是否完整，但验收项仍然停留在逐项检查。用户真正进入下一轮训练前，还需要一个明确裁决：复用回执是否已经足以放行下一轮，还是必须回到首题复用链路继续修复。

## 目标

在“首题复用回执验收卡”和“下一轮训练建议”之间增加“首题复用放行门禁”。门禁把 4 个验收项转成 4 个放行条件：分数放行、证据放行、阻断放行、下一题放行，并输出整体裁决和主行动。所有内容继续由本地规则派生，不依赖后端、账号、付费 AI 或第三方服务。

## 方案

首题复用放行门禁由 `buildPracticeSessionFirstQuestionReuseReceiptAcceptance` 派生：

- 复用回执验收卡为空时，显示等待放行状态。
- 复用回执验收卡处于回修状态时，显示“复用暂缓放行”，主行动回到复用修复入口。
- 复用回执验收卡处于 ready 状态时，显示“复用已放行”，主行动进入下一轮首题入口。
- 每个放行条件包含证据、放行检查和处理动作。
- 主行动复用验收卡入口，避免新增路由、存储或付费能力。

## 用户价值

相比只展示验收卡，放行门禁让用户知道“现在能不能带着复用证据进入下一轮”。它把免费模拟面试链路从“检查”推进到“裁决”，减少用户跳过复用证据直接刷下一题的情况。

## 验收

- Markdown 导出包含“首题复用放行门禁”，且位于“首题复用回执验收卡”之后、“下一轮训练建议”之前。
- 前端战报展示 4 个放行条件：分数放行、证据放行、阻断放行、下一题放行。
- 空会话显示等待状态，不出现 `undefined`。
- 聚焦测试、全量测试、构建和 `git diff --check` 通过。
