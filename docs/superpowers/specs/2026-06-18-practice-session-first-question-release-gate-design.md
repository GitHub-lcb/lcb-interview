# 模拟面试战报首题放行门禁设计

## 背景

战报已经能给出首题预演、首题验收尺、首题回执模板和首题回执验收卡，但这些模块仍是分散的步骤。用户完成首题链路后，还需要一个明确裁决：当前首题是否已经足够支撑进入下一轮训练，还是必须先回到本轮修复。

## 目标

在“首题回执验收卡”和“下一轮训练建议”之间增加“首题放行门禁”。该门禁聚合首题预演、验收尺、回执和回执验收 4 个放行条件，输出一个清晰结论、逐项证据和主行动，继续保持完全免费、本地规则派生、不依赖外部服务。

## 方案

首题放行门禁由现有首题链路派生：

- 首题回执验收卡为空时，显示等待状态，不生成伪裁决。
- 回执验收处于回修状态时，显示“首题暂缓放行”，主行动回到本轮修复入口。
- 回执验收处于 ready 状态时，显示“首题已放行”，主行动进入下一轮训练入口。
- 放行条件固定为 4 项：预演动作、验收口径、回执证据、放行结论。
- 每项都给出证据、放行检查和未通过处理动作，避免用户只看到笼统提示。

## 用户价值

相比只给题目列表，首题放行门禁把“能不能继续推进”变成可核验的判定。用户不会因为看到下一轮题目就盲目进入，而是先确认首题样本已经留下证据，训练节奏更接近真实教练盯盘。

## 验收

- Markdown 导出包含“首题放行门禁”，且位于“首题回执验收卡”之后。
- 前端战报展示首题放行裁决、4 个放行条件和主按钮。
- 空会话显示等待状态，不出现 `undefined`。
- 聚焦测试、全量测试、构建和 `git diff --check` 通过。
