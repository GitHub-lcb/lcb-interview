# 模拟面试战报首题复用复盘回流放行门禁设计

## 背景

首题复用复盘回流验收卡已经能检查回流动作是否可用，但验收结果还没有形成明确裁决。用户完成分数、证据、阻断和入口检查后，仍需要自己判断到底是进入下一轮训练，还是回到复用修复入口继续补材料。缺少放行门禁会让链路停在“可验收”，没有转成“能否放行”的决定。

## 目标

在“首题复用复盘回流验收卡”和“下一轮训练建议”之间增加“首题复用复盘回流放行门禁”。门禁把 4 个回流验收项转成放行裁决：分数放行、证据放行、阻断放行、入口放行。所有裁决继续由本地规则从当前战报上下文派生，不依赖后端状态、账号体系、付费 AI 或第三方服务。

## 方案

首题复用复盘回流放行门禁由 `buildPracticeSessionFirstQuestionReuseReviewHandoffAcceptance` 派生：

- 回流验收卡为空时，显示“等待首题复用复盘回流放行裁决”，提示先完成回流验收。
- 回流验收卡处于 repair 状态时，显示“回流暂缓放行”，主行动回到复用修复入口。
- 回流验收卡处于 ready 状态时，显示“回流已放行”，主行动进入下一轮首题。
- 每个门禁项包含证据、放行规则、处理动作和状态。
- 门禁复用验收卡的入口，不新增路由、存储或服务端状态。

## 用户价值

放行门禁把“验收项是否可用”压缩成“现在能不能进入下一轮”的明确判断。用户不需要在长战报里自行推理下一步，系统会直接告诉他哪些项已经放行、哪些项会阻断，并给出可点击回退动作。

## 验收

- Markdown 导出包含“首题复用复盘回流放行门禁”，且位于“首题复用复盘回流验收卡”之后、“下一轮训练建议”之前。
- 前端战报展示 4 个门禁项：分数放行、证据放行、阻断放行、入口放行。
- 每个门禁项展示“放行规则”“处理动作”“状态”。
- 空会话显示等待状态，不出现 `undefined`。
- 聚焦测试、全量测试、构建和 `git diff --check` 通过。
