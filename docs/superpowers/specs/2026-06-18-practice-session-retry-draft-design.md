# 本轮二次提交稿设计

## 背景

当前战报已经能给出复述卡、压力追问和失分禁区，但用户真正要再次提交时，还需要把这些信号合并成一版可以直接开口的回答草稿。只给“不要这样说”仍然要求用户临场组织语言，容易回到原来的低分表达。

## 目标

- 在本轮战报新增“本轮二次提交稿”。
- 基于 60 秒复述卡和失分禁区生成 1-3 份可直接重答的结构化草稿。
- 每份草稿包含结论句、证据句、边界句和收束句。
- Markdown 导出与面板展示保持一致。
- 继续纯前端本地生成，不新增后端接口，不依赖付费服务。

## 非目标

- 不生成长篇标准答案。
- 不替代用户自己的真实项目经历，只提供需要补齐的表达框架。
- 不保存草稿编辑状态，先作为战报里的可复制重答参考。

## 方案

新增 `PracticeSessionRetryDrafts` 类型和 `buildPracticeSessionRetryDrafts(queue, progress)` 构建器。

生成规则：

1. 复用 `buildPracticeSessionReplayCards` 生成二次提交稿，保证草稿来自本轮证据缺口和复述脚本。
2. 复用 `buildPracticeSessionRiskGuardrails` 判断是否处于修复态，并把主行动入口对齐到当前重答队列。
3. 每张复述卡转成一份草稿：
   - 结论句：沿用复述卡开场句，强制结论先行。
   - 证据句：沿用复述卡证据句，引导补项目规模、指标或本人动作。
   - 边界句：沿用复述卡边界句，引导补风险、监控、降级或取舍。
   - 收束句：提醒如果继续追问，就补规模、失败边界和替代方案。
4. 无复述卡时展示等待状态，引导用户先完成模拟面试。

面板位置放在“本轮失分禁区”之后、“下一轮训练”之前。这样战报从诊断、复述、自检、追问、避坑，最终落到可重答稿。

## 测试策略

- 工具测试覆盖低分场景导出二次提交稿。
- 空队列 Markdown 输出等待提示。
- 面板测试覆盖草稿内容和主行动导航。
- 提交前运行聚焦测试、完整前端测试、生产构建和 `git diff --check`。
