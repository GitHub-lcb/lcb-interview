# 本轮压力追问卡设计

## 背景

本轮战报已经能输出证据缺口、60 秒复述卡和复述验收清单，但用户完成自检后仍缺少一个更接近真实面试的压力环节。真实面试里，面试官不会只听主回答，而会连续追问“规模是多少”“失败怎么办”“为什么不用别的方案”。普通题库和单次评分很难覆盖这个环节。

## 目标

- 在本轮战报新增“本轮压力追问卡”。
- 基于当前复述卡生成 3 个连续追问，覆盖落地证据、失败边界和技术取舍。
- 每个追问包含面试官问题、暴露风险、回答口径和训练入口。
- Markdown 导出与面板展示保持一致。
- 继续纯前端本地生成，不新增后端接口，不依赖付费服务。

## 非目标

- 不替换已有“本轮追问防线”。追问防线偏全局错因防守，压力追问卡偏复述后的现场连问。
- 不引入实时语音或计时功能。
- 不保存追问完成状态，先提供静态训练脚本和跳转入口。

## 方案

新增 `PracticeSessionPressureProbes` 类型和 `buildPracticeSessionPressureProbes(queue, progress)` 构建器。

生成规则：

1. 复用 `buildPracticeSessionReplayCards` 的题目和主行动入口，保证追问卡紧跟复述卡。
2. 有复述卡时生成三类压力追问：
   - 落地证据：追问项目规模、指标、职责和验证方式。
   - 失败边界：追问异常场景、降级、监控和回滚。
   - 技术取舍：追问替代方案、成本收益和为什么这样选。
3. 优先绑定复述卡里的题目；如果复述卡不足 3 张，围绕第一张卡补齐三连问，形成连续压测。
4. 无复述卡时展示等待状态，引导用户先完成模拟面试和复述卡。

面板位置放在“本轮复述验收清单”之后、“下一轮训练”之前。用户路径为：按卡复述 -> 按清单自检 -> 用压力追问检验是否抗打 -> 再进入下一轮训练。

## 测试策略

- 工具测试覆盖低分场景导出三类压力追问。
- 空队列 Markdown 输出等待提示。
- 面板测试覆盖追问卡内容和主行动导航。
- 提交前运行聚焦测试、完整前端测试、生产构建和 `git diff --check`。
