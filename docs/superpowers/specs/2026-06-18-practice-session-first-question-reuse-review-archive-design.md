# 模拟面试战报首题复用复盘归档包设计

## 背景

首题复用复盘验收卡已经能检查分数、证据、阻断和下一题回流是否可用，但验收之后还缺少归档视图。用户即使通过了复用复盘验收，也可能不知道哪些内容要带入下一轮训练，导致复用链路在“验收完成”后无法沉淀成可继续调用的训练证据。

## 目标

在“首题复用复盘验收卡”和“下一轮训练建议”之间增加“首题复用复盘归档包”。归档包把复盘验收结果整理成 4 个下一轮输入：复用分数快照、复用证据归档、阻断回收结论、下一轮回流种子。所有内容继续由本地规则派生，不依赖后端、账号、付费 AI 或第三方服务。

## 方案

首题复用复盘归档包由 `buildPracticeSessionFirstQuestionReuseReviewAcceptance` 派生：

- 复盘验收卡为空时，显示等待归档状态。
- 复盘验收卡处于 repair 状态时，显示“回修复用复盘归档包”，强调归档未放行项的修复输入。
- 复盘验收卡处于 ready 状态时，显示“首题复用复盘归档包”，强调把通过验收的复用复盘保存成下一轮输入。
- 每个归档项包含来源、归档内容、下一轮用途和丢失风险。
- 主行动复用验收卡入口，避免新增路由、存储或付费能力。

## 用户价值

相比只有复盘验收卡，归档包会告诉用户“这次复用复盘最终留下了什么”。它把免费模拟面试链路从检查动作推进到可复用资产，让下一轮训练可以承接已经验证过的分数、证据、阻断和题序信息。

## 验收

- Markdown 导出包含“首题复用复盘归档包”，且位于“首题复用复盘验收卡”之后、“下一轮训练建议”之前。
- 前端战报展示 4 个归档项：复用分数快照、复用证据归档、阻断回收结论、下一轮回流种子。
- 空会话显示等待状态，不出现 `undefined`。
- 聚焦测试、全量测试、构建和 `git diff --check` 通过。
