# 模拟面试战报首题复用复盘回流验收卡设计

## 背景

首题复用复盘回流清单已经能把归档包拆成下一轮动作，但动作清单本身还缺少提交前验收。如果用户只是看到“分数回流、证据带入、阻断处置、下一题启动”，仍可能不知道这些动作是否已经足够清楚，是否可以带入下一轮训练，或者是否应该回到复用修复入口继续补证据。

## 目标

在“首题复用复盘回流清单”和“下一轮训练建议”之间增加“首题复用复盘回流验收卡”。验收卡把 4 个回流动作转成可判定标准：分数可核验、证据可带入、阻断可处置、入口可执行。所有判断继续由本地规则从当前战报上下文派生，不依赖后端状态、账号体系、付费 AI 或第三方服务。

## 方案

首题复用复盘回流验收卡由 `buildPracticeSessionFirstQuestionReuseReviewHandoff` 派生：

- 回流清单为空时，显示“等待验收首题复用复盘回流”，提示先生成回流清单。
- 回流清单处于 repair 状态时，显示“回修复用复盘回流待验收”，强调先补齐不可执行项。
- 回流清单处于 ready 状态时，显示“首题复用复盘回流待验收”，强调进入下一轮前做最后检查。
- 每个验收项包含验收目标、通过信号、缺失风险和补救动作。
- 主行动复用回流清单入口，避免新增路由、存储或复杂状态。

## 用户价值

回流验收卡把“我要做什么”继续推进到“我怎么知道已经能做”。它降低用户在免费模拟面试里自查的成本，让复用复盘不会停在任务列表，而是形成可验收、可回退、可继续训练的闭环。

## 验收

- Markdown 导出包含“首题复用复盘回流验收卡”，且位于“首题复用复盘回流清单”之后、“下一轮训练建议”之前。
- 前端战报展示 4 个验收项：分数可核验、证据可带入、阻断可处置、入口可执行。
- 每个验收项展示“通过信号”“缺失风险”“补救动作”。
- 空会话显示等待状态，不出现 `undefined`。
- 聚焦测试、全量测试、构建和 `git diff --check` 通过。
