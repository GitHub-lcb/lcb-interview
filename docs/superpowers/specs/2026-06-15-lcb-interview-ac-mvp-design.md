# LCB Interview A+C MVP Design

## 背景

用户目标是让本站明显强于 mianshiya.com，而不是只做一个题库克隆。公开页面显示，面试鸭当前优势集中在题库规模、刷题路线、开始面试、真实面经、社区/会员、多端生态等方面；其 about 页面也强调 1w+ 高频题、原创题解、持续更新、沉浸式学习和移动端使用。参考页面：

- https://www.mianshiya.com/
- https://www.mianshiya.com/category/hot
- https://www.mianshiya.com/about

本项目已有 Spring Boot + React 题库浏览、题目详情、搜索、管理后台和 AI 生成/审核雏形。第一阶段不追求题量全面赶超，而是选择更容易形成差异的两条路线：

- A：答案质量碾压。让每道题不仅有答案，还能告诉用户怎么答、怎么被追问、哪里容易答错、如何结合项目经验。
- C：个性化备考系统。让用户不是漫无目的刷题，而是看到目标岗位、今日计划、掌握度和薄弱项。

## 第一阶段目标

把当前题库站升级为“高质量答案 + 本地个性化备考”的可用 MVP。用户打开首页后能看到岗位备考状态，进入题目详情后能看到结构化回答模块，并能把题加入今日计划、标记掌握程度，回到首页看到计划和弱点变化。

第一阶段明确不做：

- 不做完整用户账号体系。
- 不做后端用户进度表。
- 不做社区、评论、会员、支付。
- 不做 AI 模拟面试会话。
- 不重构整个前端设计系统。

## 产品设计

### 1. 首页升级为备考工作台

首页从“热门题库 + 热门题目”升级为“备考工作台 + 题库入口”。首屏保留题库入口，但增加明确的个人化信息：

- 目标岗位：默认 Java 后端，后续可在前端本地切换。
- 冲刺周期：默认 21 天。
- 今日计划：展示 3 到 5 道推荐题。
- 掌握度：按本地答题状态计算已掌握比例。
- 弱点雷达：按分类展示薄弱项，例如 Java 并发、JVM、MySQL。

本阶段用 `localStorage` 保存用户选择和题目状态，避免账号体系拖慢上线。数据结构只面向前端：

```ts
interface StudyProgress {
  targetRole: string
  sprintDays: number
  questionStates: Record<number, QuestionStudyState>
  dailyPlan: number[]
}

interface QuestionStudyState {
  status: 'new' | 'learning' | 'mastered' | 'weak'
  addedToPlan: boolean
  lastReviewedAt?: string
  reviewCount: number
}
```

### 2. 题目详情升级为答案质量页

题目详情页保持当前 Markdown 渲染能力，但把已有字段组织成面试场景可用的模块：

- 30 秒口径：来自 `summary`，用于快速背诵。
- 标准回答：来自 `content`。
- 原理深挖：来自 `principle`。
- 对比分析：来自 `comparison`。
- 项目落地：来自 `projectExp`。
- 适用场景：来自 `scenario`。
- 风险误区：来自 `risk`。
- 代码示例：来自 `codeExamples`。
- 图解：来自 `diagrams`。

新增前端派生展示，不要求第一阶段改数据库：

- 答案质量评分：基于字段完整度计算 0-100。
- 面试官追问：先用静态规则从分类/标题/字段生成 3 个追问，后续可由 AI 生成。
- 不要这么答：优先使用 `risk` 摘要，没有则显示通用提示。
- 操作按钮：加入今日计划、标记薄弱、标记掌握。

### 3. 题库列表和搜索结果增加学习状态

题目列表卡片增加本地状态标识：

- 未学
- 学习中
- 薄弱
- 已掌握
- 已加入今日计划

筛选仍沿用现有后端接口，状态筛选第一阶段只在前端当前页内生效，不新增后端 API。

### 4. 本地推荐策略

今日计划生成策略保持简单、可解释：

1. 优先选 `weak` 状态题目。
2. 其次选浏览过但未掌握题目。
3. 再按热门题和分类入口补足。
4. 每日计划默认最多 5 道。

弱点雷达按题目 `categoryId/categoryName` 汇总：

- `weak` 权重最高。
- `learning` 次之。
- `mastered` 降低薄弱权重。

第一阶段不追求精确算法，重点是让用户感到“系统知道我该刷什么”。

## 技术设计

### 前端文件边界

- 新增 `frontend/src/utils/studyProgress.ts`
  - 负责 localStorage 读写、默认值、状态更新、推荐计划计算。
- 新增 `frontend/src/components/StudyDashboard.tsx`
  - 首页备考工作台。
- 新增 `frontend/src/components/StudyStatusBadge.tsx`
  - 列表和详情复用的学习状态标识。
- 修改 `frontend/src/pages/Home/index.tsx`
  - 接入备考工作台，保留题库入口和热门题。
- 修改 `frontend/src/pages/QuestionDetail/index.tsx`
  - 接入学习操作按钮和答案质量侧栏。
- 修改 `frontend/src/pages/QuestionDetail/ContentView.tsx`
  - 将内容模块化展示。
- 修改 `frontend/src/pages/QuestionList/index.tsx`
  - 卡片显示学习状态和加入计划按钮。
- 修改 `frontend/src/types.ts`
  - 增加学习状态类型。

### 后端边界

第一阶段尽量不改后端业务表。已有 `QuestionVO` 字段已能支撑答案质量页。必要时只补充测试或小型 VO 字段，但优先用前端派生信息。

### 兼容策略

- 如果题目缺少 `summary`，前端用 `content` 第一段生成“快速回答”显示。
- 如果缺少 `risk`，显示通用误区提示。
- 如果 `codeExamples/diagrams` 不是合法 JSON，前端降级为原始文本或隐藏模块，不阻塞页面。
- `localStorage` 解析失败时重置为默认状态。

## 视觉方向

整体采用“高密度但清晰的备考工作台”风格，不做营销首页。界面应像专业学习工具，而不是普通内容列表：

- 首页首屏直接显示学习状态，不做大 hero。
- 题目详情页使用主内容 + 右侧质量/操作栏。
- 卡片半径控制在 8px 左右，避免过度圆角。
- 色彩使用中性色底 + 蓝/绿/橙状态色，不做单一紫蓝渐变。
- 字号克制，工具型信息优先扫描效率。

## 验收标准

- 首页能看到目标岗位、今日计划、掌握度、弱点雷达。
- 用户能在题目详情页加入今日计划、标记薄弱、标记掌握。
- 题目详情页展示 30 秒口径、标准回答、原理、项目、误区等模块。
- 题目列表能展示本地学习状态。
- 刷新页面后状态仍存在。
- localStorage 损坏时页面不崩溃。
- `npm run build` 通过。
- 已有后端测试继续通过。

## 风险

- 本地进度不跨设备。接受，因为第一阶段目标是快速做出差异，第二阶段再接账号体系。
- AI 生成内容质量不稳定。第一阶段先提升呈现结构和审核体验，不依赖实时 AI。
- 当前前端中文文本存在历史乱码风险。实施时应优先改动涉及页面的可见文案，必要时统一修复相关文件编码。
- 首包已较大。实现时避免新增重型依赖，优先原生 React + Ant Design。

## 后续阶段

第二阶段建议：

- 后端用户体系和进度表。
- 收藏、错题本、复习提醒。
- AI 模拟面试：基于题目追问、回答评分、简历项目追问。
- AI 内容质检：答案完整度、事实错误、过时技术自动标记。
- 面经/公司题单：把题目和真实面试场景连接起来。

## 自审记录

- 规格覆盖了用户批准的 A+C 范围。
- 第一阶段明确排除了账号、社区、会员、AI 模拟面试，避免范围失控。
- 数据策略优先复用现有 `QuestionVO`，降低后端改动。
- 当前工具政策不允许在用户未明确要求子代理的情况下主动派生 spec reviewer，因此本规格未运行子代理审查。
