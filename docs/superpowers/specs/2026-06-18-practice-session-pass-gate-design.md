# 本轮通过门槛设计

## 背景

当前模拟面试战报已经能给出能力雷达、面试官判定、行动优先级、证据缺口、60 秒复述卡、复述验收清单、压力追问、失分禁区和二次提交稿。用户仍需要一个更直接的结论：这轮到底能不能进入下一轮，不能的话先卡在哪一道门槛。

## 目标

新增“本轮通过门槛”模块，把本轮战报里的分散信号压缩成可执行的通关判断。模块必须做到：

- 明确展示本轮是否“可以推进”或“暂缓推进”。
- 展示 4 个硬门槛：全题完成、平均分达标、弱项清零、二次提交稿就绪。
- 每个门槛展示目标、当前状态和修复动作。
- Markdown 导出包含同样信息，方便复制到复盘记录。
- 空队列或未开口时给出等待态，不制造虚假的通过结论。

## 设计

通过门槛由前端纯函数 `buildPracticeSessionPassGate` 生成，不新增后端依赖。它复用现有的 `buildPracticeSessionReport`、`buildPracticeSessionAbilityRadar` 和 `buildPracticeSessionRetryDrafts`：

- 队列为空或未答题：状态为 `empty`，主行动指向现有开始练习入口。
- 已答但仍有未完成题：全题完成门槛阻塞，主行动继续未答题。
- 平均分低于 80：平均分门槛阻塞，主行动走当前战报的修复入口。
- 存在弱题或能力雷达未稳定：弱项清零门槛阻塞，主行动走修复入口。
- 二次提交稿未生成：提交稿门槛阻塞，主行动引导先建立开口样本。
- 全部门槛满足：状态为 `ready`，主行动进入下一轮训练。

## UI

模块放在“本轮二次提交稿”和“下一轮训练”之间，因为它是重答稿之后、进入下一轮之前的最后闸门。视觉上用紧凑的门槛列表，不使用大面积卡片嵌套；每条门槛包含状态徽标、目标、当前值和动作提示。

## 测试

- 工具层测试 Markdown 包含“本轮通过门槛”、4 个门槛和主行动。
- 空队列测试等待态。
- 组件测试断言模块渲染、门槛标签展示、按钮导航。
- 运行 `npm run test -- practiceSessionReport PracticeSessionReportPanel`、`npm run test` 和 `npm run build`。
