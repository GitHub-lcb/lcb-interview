# 面试错因修复计划设计

## 背景

面试错因本已经能把历史模拟面试中的低分维度和未开口薄弱题聚合出来，但用户看到错因后仍需要自己判断“先做哪一步、每一步练什么、什么时候算完成”。为了把错因本从诊断升级为闭环训练，需要在错因本下方给出短周期、可执行的修复计划。

## 目标

- 基于 `InterviewMistakeLedger` 生成 2-3 个修复步骤。
- 每个步骤都给出训练动作、原因、预计耗时和跳转入口。
- 优先修复最高优先级错因，再安排巩固和加压。
- 保持完全免费和本地计算，不新增后端接口。
- 空状态时引导用户先完成模拟面试，不制造虚假的修复计划。

## 非目标

- 不做跨设备同步。
- 不引入日历、提醒或账号体系。
- 不改变现有 `/practice?queue=...` 队列协议。

## 设计方案

新增 `buildInterviewRecoveryPlan(ledger)` 工具函数：

1. 当错因本为空时，返回一条“建立第一条面试记录”的步骤。
2. 当存在高风险错因时：
   - 第一步：针对首要错因进行回炉训练，使用首要错因的队列链接。
   - 第二步：针对同一批题补齐结构、场景或风险表达。
   - 第三步：使用同一队列做一次高压追问复测。
3. 当错因本稳定时：
   - 生成“连续追问”“边界场景”“复盘沉淀”三步加压计划。

新增类型：

- `InterviewRecoveryStep`
- `InterviewRecoveryPlan`

在 `InterviewMistakeLedgerPanel` 中展示修复计划：

- 放在错因卡片下方。
- 使用紧凑步骤列表，每个步骤有序号、耗时、原因和按钮。
- 首步骤按钮使用 primary，其余步骤使用 default。

## 行为规则

- 修复步骤最多 3 个，避免学习计划页膨胀。
- 所有跳转必须复用错因项的 `to`，保持训练队列一致。
- 首要错因没有题目 ID 时降级到 `/practice`。
- 稳定状态不再提示“错因”，改用“加压计划”措辞。

## 测试策略

- 单元测试覆盖空状态、高风险错因、薄弱未开口、稳定加压四种路径。
- 验证步骤数量、首步骤跳转、总耗时和稳定状态文案。
- 前端全量测试和构建验证页面接入。
