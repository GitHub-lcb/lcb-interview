# 模拟面试战报首题复用复盘回流放行回执验收卡设计

## 背景

首题复用复盘回流放行回执已经能把门禁裁决整理成可携带的交接记录，但回执生成后还缺少最后一道验收。用户看到“放行结论、随身证据、阻断处理、下一轮入口”后，仍需要自己判断这份回执能不能真的带入下一轮首题。缺少验收卡会让链路从“生成回执”直接跳到“下一轮训练”，中间没有确认回执质量的步骤。

## 目标

在“首题复用复盘回流放行回执”和“下一轮训练建议”之间新增“首题复用复盘回流放行回执验收卡”。验收卡把 4 个回执项转成可判定标准：结论可判、证据可带、阻断可控、入口可执行。所有内容继续由当前战报本地规则派生，不依赖后端状态、账号系统、付费 AI 或第三方服务。

## 方案

回执验收卡由 `buildPracticeSessionFirstQuestionReuseReviewHandoffReleaseReceipt` 派生：

- 回执为空时，显示“等待验收首题复用复盘回流放行回执”，提示先生成放行回执。
- 回执阻断时，显示“回流放行回执待验收”，主行动回到回流修复入口。
- 回执通过时，显示“回流放行回执已验收”，主行动进入下一轮首题。
- 每个验收项包含验收目标、通过信号、缺失风险和补救动作，直接对应回执中的 4 个交接项。
- 验收卡复用回执的主行动入口，不新增路由、持久化或服务端状态。

## 用户价值

用户可以在进入下一轮训练前确认：这份回执是不是有明确结论、可追问证据、可控阻断和可点击入口。它把“我生成了回执”提升为“这份回执可以交接”，减少带着半成品进入下一轮导致的重复返工。

## 验收

- Markdown 导出包含“首题复用复盘回流放行回执验收卡”，且位于“首题复用复盘回流放行回执”之后、“下一轮训练建议”之前。
- 前端战报展示 4 个验收项：结论可判、证据可带、阻断可控、入口可执行。
- 每个验收项展示“通过信号”“缺失风险”“补救动作”。
- 空会话显示等待状态，不出现 `undefined`。
- 聚焦测试、全量测试、构建和 `git diff --check` 通过。
